# Phase 6.1: Visibility & Sharing - Research

**Researched:** 2026-03-20
**Domain:** JPA entity authorization, share tables, visibility-scoped queries
**Confidence:** HIGH

## Summary

This phase adds visibility controls (PRIVATE/PUBLIC/SHARED) to companies and jobs, plus a share management system allowing users to share specific items with other users by email. The domain is well-understood: it requires a visibility enum column on existing entities, a join table for share grants, and modified repository queries that factor in visibility and share permissions.

No new libraries are needed. The existing Spring Boot 4.0.4 + JPA + PostgreSQL stack handles everything. The complexity lies in correct query design (OR-based visibility filtering for browse/shared views), authorization checks in services (owner vs shared user with VIEW/EDIT permission), and ensuring zero behavior change for existing data (default PRIVATE).

**Primary recommendation:** Add a `visibility` column to companies and jobs tables, create a `share_grants` join table with permission levels, and build a dedicated `VisibilityService` that encapsulates all visibility/sharing logic separate from existing services. Modify existing service methods to check share grants for read/edit access.

### Pattern 1: Visibility Enum on Entity
**What:** Add a `visibility` column (VARCHAR, default PRIVATE) directly on the companies and jobs tables.
**When to use:** When visibility is a property of the entity itself, not a separate concern.
**Example:**
```kotlin
// In enums.kt
enum class Visibility { PRIVATE, PUBLIC, SHARED }
enum class SharePermission { VIEW, EDIT }

// In CompanyEntity.kt - add field
@Column(nullable = false, length = 20)
@Enumerated(EnumType.STRING)
var visibility: Visibility = Visibility.PRIVATE
```

### Pattern 2: Share Grants Join Table
**What:** A dedicated table linking a resource (company/job) to a target user with a permission level.
**When to use:** For tracking who has access to what with what permission.
**Example:**
```kotlin
@Entity
@Table(name = "share_grants")
class ShareGrantEntity(
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    @Column(updatable = false, nullable = false)
    val id: UUID? = null,

    @Column(name = "owner_id", nullable = false)
    val ownerId: UUID,

    @Column(name = "target_user_id", nullable = false)
    val targetUserId: UUID,

    @Column(name = "resource_type", nullable = false, length = 20)
    @Enumerated(EnumType.STRING)
    val resourceType: ShareResourceType,

    @Column(name = "resource_id", nullable = false)
    val resourceId: UUID,

    @Column(nullable = false, length = 20)
    @Enumerated(EnumType.STRING)
    var permission: SharePermission = SharePermission.VIEW,

    @Column(name = "created_at", updatable = false, nullable = false)
    val createdAt: Instant = Instant.now()
)

enum class ShareResourceType { COMPANY, JOB }
```

### Pattern 3: Visibility-Aware Queries with JPQL
**What:** Repository methods that return items visible to a user (own + public + shared).
**When to use:** For browse/list endpoints that cross user boundaries.
**Example:**
```kotlin
// Public browse - no auth required for reading, but shows only PUBLIC items
@Query("""
    SELECT c FROM CompanyEntity c
    WHERE c.visibility = 'PUBLIC'
    AND c.archived = false
    AND (:name IS NULL OR LOWER(c.name) LIKE LOWER(CONCAT('%', CAST(:name AS string), '%')))
""")
fun findPublic(name: String?, pageable: Pageable): Page<CompanyEntity>

// Shared with me - items where I am the target user in share_grants
@Query("""
    SELECT c FROM CompanyEntity c
    WHERE c.archived = false
    AND EXISTS (
        SELECT 1 FROM ShareGrantEntity sg
        WHERE sg.resourceType = 'COMPANY'
        AND sg.resourceId = c.id
        AND sg.targetUserId = :userId
    )
""")
fun findSharedWithUser(userId: UUID, pageable: Pageable): Page<CompanyEntity>
```

### Pattern 4: Authorization Check in Service Layer
**What:** Service methods check ownership OR share grant before allowing access.
**When to use:** For getById, update, delete operations that need to respect sharing.
**Example:**
```kotlin
// In CompanyService - modified getById
fun getById(id: UUID, userId: UUID): CompanyResponse {
    val entity = companyRepository.findById(id).orElse(null)
        ?: throw NotFoundException("Company not found")

    // Owner always has full access
    if (entity.userId == userId) return entity.toResponse()

    // Public items are viewable by anyone
    if (entity.visibility == Visibility.PUBLIC) return entity.toResponse()

    // Check share grant
    val grant = shareGrantRepository.findByResourceTypeAndResourceIdAndTargetUserId(
        ShareResourceType.COMPANY, id, userId
    ) ?: throw NotFoundException("Company not found")  // 404, not 403, to avoid leaking existence

    return entity.toResponse()
}
```

### Pattern 5: Separate Public Browse Endpoints
**What:** Dedicated endpoints for browsing public content, separate from user's own list.
**When to use:** To keep existing list endpoints unchanged (backward compatible) while adding new public views.
**Example:**
```kotlin
// New endpoints - don't modify existing /api/companies GET
@GetMapping("/api/public/companies")     // Browse all public companies
@GetMapping("/api/shared/companies")     // View companies shared with me
@GetMapping("/api/companies/{id}/shares") // Manage shares on my company
```

### Anti-Patterns to Avoid
- **Modifying existing list endpoints to include public/shared items:** Keep existing endpoints for the owner's own data. Add separate endpoints for public browse and shared-with-me views.
- **Using 403 for unauthorized access to shared resources:** Return 404 to avoid leaking resource existence to unauthorized users.
- **Putting authorization logic in controllers:** Keep it in the service layer where it can be tested and reused.
- **Cascading visibility to child entities:** Requirements say visibility is on companies and jobs only. Don't add visibility to applications, interviews, or documents.

## Common Pitfalls

### Pitfall 1: Breaking Existing Behavior with Default Visibility
**What goes wrong:** Existing companies/jobs have NULL visibility, breaking queries.
**Why it happens:** Adding a non-null column without a DEFAULT or data migration.
**How to avoid:** Migration must add column with DEFAULT 'PRIVATE' and backfill existing rows: `ALTER TABLE companies ADD COLUMN visibility VARCHAR(20) NOT NULL DEFAULT 'PRIVATE'`. PostgreSQL handles this efficiently for non-null defaults.
**Warning signs:** Existing integration tests fail after migration.

### Pitfall 2: N+1 Queries on Share Grants
**What goes wrong:** Checking share grants per-item in a list query.
**Why it happens:** Calling shareGrantRepository.find() inside a loop for each item.
**How to avoid:** Use EXISTS subquery in JPQL (already demonstrated above) or batch-fetch grants for a page of results.
**Warning signs:** Slow list queries, excessive SQL in logs.

### Pitfall 3: Leaking Resource Existence via 403
**What goes wrong:** Returning 403 Forbidden when a user tries to access someone else's PRIVATE resource.
**Why it happens:** Natural to throw ForbiddenException for unauthorized access.
**How to avoid:** Return 404 Not Found for any resource the user shouldn't know about. Only use 403 when the user knows the resource exists (e.g., they had access and it was revoked, or they're trying to edit with VIEW permission).
**Warning signs:** API returns 403 for random UUIDs, confirming resource existence.

### Pitfall 4: Forgetting to Check Visibility on Update/Delete
**What goes wrong:** A shared user with EDIT permission can delete the resource.
**Why it happens:** The existing update/delete pattern only checks userId == owner.
**How to avoid:** EDIT permission grants update rights only. Delete is always owner-only. Make this explicit in the service.
**Warning signs:** Shared users can archive/delete items they don't own.

### Pitfall 5: Orphaned Share Grants
**What goes wrong:** Share grants remain after the resource is deleted/archived.
**Why it happens:** No cascade or cleanup when visibility changes to PRIVATE.
**How to avoid:** When visibility changes from SHARED to PRIVATE or PUBLIC, delete all share grants for that resource. When a resource is archived, clean up grants.
**Warning signs:** share_grants table grows indefinitely, "shared with me" shows items that are now private.

### Pitfall 6: Self-Sharing
**What goes wrong:** User shares an item with their own email.
**Why it happens:** No validation that target email != owner email.
**How to avoid:** Check in the service that the resolved target user ID != owner ID. Return 400 Bad Request.
**Warning signs:** User appears in their own share list.

### Pitfall 7: CAST for Null Parameters in JPQL
**What goes wrong:** PostgreSQL throws `could not determine data type of parameter` for null string parameters.
**Why it happens:** Known PostgreSQL + Hibernate issue with nullable parameters.
**How to avoid:** Use `CAST(:param AS string)` pattern already established in this codebase (CompanyRepository, JobRepository).
**Warning signs:** DataAccessException when calling queries with null optional parameters.

### ShareService Pattern
```kotlin
@Service
@Transactional
class ShareService(
    private val shareGrantRepository: ShareGrantRepository,
    private val userRepository: UserRepository,
    private val companyRepository: CompanyRepository,
    private val jobRepository: JobRepository
) {
    fun shareCompany(companyId: UUID, request: CreateShareRequest, ownerId: UUID): ShareGrantResponse {
        val company = companyRepository.findByIdAndUserId(companyId, ownerId)
            ?: throw NotFoundException("Company not found")

        val targetUser = userRepository.findByEmail(request.email)
            ?: throw NotFoundException("User not found with that email")

        if (targetUser.id == ownerId) throw BadRequestException("Cannot share with yourself")

        // Auto-set visibility to SHARED if currently PRIVATE
        if (company.visibility == Visibility.PRIVATE) {
            company.visibility = Visibility.SHARED
            company.updatedAt = Instant.now()
            companyRepository.save(company)
        }

        val existing = shareGrantRepository.findByResourceTypeAndResourceIdAndTargetUserId(
            ShareResourceType.COMPANY, companyId, targetUser.id!!
        )
        if (existing != null) throw ConflictException("Already shared with this user")

        val grant = ShareGrantEntity(
            ownerId = ownerId,
            targetUserId = targetUser.id!!,
            resourceType = ShareResourceType.COMPANY,
            resourceId = companyId,
            permission = request.permission
        )
        return shareGrantRepository.save(grant).toResponse(targetUser.email)
    }
}
```

### Checking Share Permission for Edits
```kotlin
// In CompanyService - modified update method
fun update(id: UUID, request: UpdateCompanyRequest, userId: UUID): CompanyResponse {
    val entity = companyRepository.findById(id).orElse(null)
        ?: throw NotFoundException("Company not found")

    if (entity.userId == userId) {
        // Owner - full access, proceed with update
    } else {
        // Check for EDIT share grant
        val grant = shareGrantRepository.findByResourceTypeAndResourceIdAndTargetUserId(
            ShareResourceType.COMPANY, id, userId
        ) ?: throw NotFoundException("Company not found")

        if (grant.permission != SharePermission.EDIT) {
            throw ForbiddenException("You only have view access to this company")
        }
    }

    entity.name = request.name
    // ... rest of update
    return companyRepository.save(entity).toResponse()
}
```

## Open Questions

1. **Should public browse require authentication?**
   - What we know: Public content is visible to anyone by definition. Current security config requires auth for all `/api/**`.
   - What's unclear: Whether unauthenticated users should browse public content.
   - Recommendation: Require authentication for public browse endpoints. The app is for job seekers who are logged in. Add `/api/public/companies` and `/api/public/jobs` but keep them behind auth. This simplifies implementation (no changes to SecurityConfig) and avoids abuse concerns.

2. **Should changing visibility from SHARED to PRIVATE auto-delete share grants?**
   - What we know: Orphaned grants are confusing. User intent when changing to PRIVATE is to revoke all access.
   - Recommendation: Yes -- when visibility changes to PRIVATE, delete all share_grants for that resource. When changing to PUBLIC, keep grants (PUBLIC is superset of SHARED).

3. **What happens when a shared user's account is deleted?**
   - What we know: No user deletion feature exists yet.
   - Recommendation: Add FK constraint with ON DELETE CASCADE from share_grants.target_user_id to users.id. Future-proofs without additional work.
