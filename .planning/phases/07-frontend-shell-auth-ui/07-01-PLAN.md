<!-- Backend auth API endpoints (integration target) -->

POST /api/auth/register  { email: string, password: string } -> 201 { message: string }
POST /api/auth/login     { email: string, password: string } -> 200 { accessToken: string, expiresIn: number, tokenType: string } + Set-Cookie: refresh_token (httpOnly, path=/api/auth/refresh)
POST /api/auth/refresh   (reads refresh_token cookie)        -> 200 { accessToken: string, expiresIn: number, tokenType: string } + Set-Cookie: refresh_token (rotated)
POST /api/auth/logout    (Authorization header + cookie)     -> 200 { message: string } + clear refresh_token cookie
GET  /api/auth/verify?token={token}                          -> 200 { message: string }
POST /api/auth/password-reset       { email: string }        -> 200 { message: string }
POST /api/auth/password-reset/confirm { token: string, newPassword: string } -> 200 { message: string }

Error response format: { status: number, error: string, message: string, path: string, timestamp: string, fieldErrors?: Record<string, string> }

CORS: localhost:3000 allowed with credentials

<!-- Better Auth integration approach (per user decision in CONTEXT.md) -->

Better Auth is the auth framework. It runs on the Next.js side as a session/proxy layer wrapping the backend JWT API.

Architecture:
- `lib/auth.ts` (server): Better Auth config with `emailAndPassword: { enabled: true }`. Uses hooks to intercept sign-in/sign-up and proxy to the backend. Runs in stateless cookie-cache mode (no Better Auth database tables needed -- the backend owns user data).
- `lib/auth-client.ts` (client): `createAuthClient()` from `better-auth/react` providing `authClient.signIn.email()`, `authClient.signUp.email()`, `authClient.signOut()`, `authClient.useSession()`.
- `app/api/auth/[...all]/route.ts`: Better Auth's Next.js route handler via `toNextJsHandler()`.
- `api-client.ts`: Kept for non-auth API calls (companies, jobs, etc. in Phase 8+). Still handles JWT attach + 401 refresh.

IMPORTANT: Better Auth is designed to own its own database. Since we are wrapping an existing backend, we need to configure Better Auth in cookie-cache/stateless mode and use hooks/custom logic to proxy auth operations. If Better Auth's hooks do not cleanly support proxying signIn.email to an external API, fall back to:
- Use Better Auth for session cookie management only (useSession, getSessionCookie)
- Create custom Next.js API routes under /api/auth/ that call the backend and then create/destroy Better Auth sessions programmatically
- This hybrid approach still uses Better Auth's client hooks and session management while delegating actual auth to the backend
