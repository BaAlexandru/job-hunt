**A. Fix apiClient for FormData uploads** (`frontend/lib/api-client.ts`):

Modify the `apiClient` function so it only sets `Content-Type: application/json` when the body is NOT a FormData instance. When body is FormData, omit Content-Type entirely so the browser sets the multipart boundary automatically.

```typescript
const headers: HeadersInit = {
  ...(!(options.body instanceof FormData) && { "Content-Type": "application/json" }),
  ...options.headers,
}
```

Keep the rest of apiClient unchanged. Also export `API_BASE` so hooks can use it for FormData uploads that bypass apiClient if needed:
```typescript
export const API_BASE = process.env.NEXT_PUBLIC_API_URL || "http://localhost:8080/api"
```

**C. Create TanStack Query hooks** following the pattern from RESEARCH.md Pattern 1:

Each hooks file exports:
- Query key factory (e.g., `applicationKeys`)
- `useX(filters?)` for list query (paginated)
- `useX(id)` for single entity query
- `useCreateX()` mutation
- `useUpdateX()` mutation
- `useDeleteX()` mutation

Specific hooks per domain:

`frontend/hooks/use-applications.ts`:
- `useApplications(filters: { status?: string, companyId?: string, search?: string, fromDate?: string, toDate?: string, page?: number, size?: number, sort?: string, includeArchived?: boolean })` -> GET /applications with query params
- `useApplication(id)` -> GET /applications/{id}
- `useCreateApplication()` -> POST /applications
- `useUpdateApplication()` -> PUT /applications/{id}
- `useUpdateApplicationStatus()` with optimistic update pattern from RESEARCH.md Pattern 2 -> PATCH /applications/{id}/status
- `useDeleteApplication()` -> DELETE /applications/{id}
- `useApplicationTransitions(id)` -> GET /applications/{id}/transitions
- `applicationKeys` factory

`frontend/hooks/use-companies.ts`:
- `useCompanies()` -> GET /companies (no pagination, returns array)
- `useCompany(id)` -> GET /companies/{id}
- `useCreateCompany()` -> POST /companies
- `useUpdateCompany()` -> PUT /companies/{id}
- `useDeleteCompany()` -> DELETE /companies/{id}
- `companyKeys` factory

`frontend/hooks/use-jobs.ts`:
- `useJobs(filters: { companyId?: string, page?: number, size?: number })` -> GET /jobs with query params
- `useJob(id)` -> GET /jobs/{id}
- `useCreateJob()` -> POST /jobs
- `useUpdateJob()` -> PUT /jobs/{id}
- `useDeleteJob()` -> DELETE /jobs/{id}
- `jobKeys` factory

`frontend/hooks/use-interviews.ts`:
- `useInterviews(applicationId)` -> GET /applications/{applicationId}/interviews
- `useCreateInterview()` -> POST /interviews
- `useUpdateInterview()` -> PUT /interviews/{id}
- `useDeleteInterview()` -> DELETE /interviews/{id}
- Interview notes: `useInterviewNotes(interviewId)` -> GET /interviews/{interviewId}/notes
- `useCreateInterviewNote()` -> POST /interviews/{interviewId}/notes
- `interviewKeys` factory

`frontend/hooks/use-documents.ts`:
- `useDocuments(filters: { category?: string, applicationId?: string })` -> GET /documents with query params
- `useUploadDocument()` -> POST /documents with FormData (use fetch directly with credentials: "include", NOT apiClient -- no Content-Type header)
- `useDeleteDocument()` -> DELETE /documents/{id}
- `useDownloadDocument(id)` returns download URL: `${API_BASE}/documents/${id}/download`
- `documentKeys` factory

Application notes hooks (include in use-applications.ts):
- `useApplicationNotes(applicationId)` -> GET /applications/{applicationId}/notes
- `useCreateApplicationNote()` -> POST /applications/{applicationId}/notes
- `useUpdateApplicationNote()` -> PUT /applications/{applicationId}/notes/{noteId}

Timeline hook (include in use-applications.ts):
- `useTimeline(applicationId)` -> GET /applications/{applicationId}/timeline

`frontend/hooks/use-view-preference.ts`:
- `useViewPreference(key: string, defaultView: "board" | "list" = "board")` returns `[view, setView]`
- Uses localStorage with SSR safety (check `typeof window !== "undefined"`)
- Key for applications: `"jobhunt:applications-view"`

All mutations should call `queryClient.invalidateQueries()` on success to refetch affected data. The status update mutation specifically must use the optimistic update pattern with `onMutate` (cancel queries, snapshot previous, optimistically update cache), `onError` (rollback), `onSettled` (invalidate).
