  <action>
Modify `authenticateFromSession` in BetterAuthSessionFilter.kt to handle the case where a valid Better Auth session exists but no matching backend `users` row is found.

**Approach:** Change the single JOIN query into a two-step process:

1. First query: Validate the Better Auth session and get the Better Auth user's email:
```sql
SELECT u.email
FROM session s
JOIN "user" u ON u.id = s."userId"
WHERE s.token = ? AND s."expiresAt" > NOW()
```

2. Look up the backend user by email. If not found, auto-create one:
   - Use `jdbcTemplate.queryForMap` to check `SELECT id, email, enabled FROM users WHERE email = ?`
   - If `EmptyResultDataAccessException` is thrown (no row), INSERT a new user:
     ```sql
     INSERT INTO users (id, email, password, role, enabled, created_at, updated_at)
     VALUES (gen_random_uuid(), ?, '', 'USER', true, NOW(), NOW())
     RETURNING id, email, enabled
     ```
   - The password is empty string because Better Auth users authenticate via session cookies, not JWT password flow.
   - Set `enabled = true` because Better Auth already handled email verification.

3. Continue with the existing authentication setup logic (create AppUserDetails, set SecurityContext).

Import `org.springframework.dao.EmptyResultDataAccessException` for the catch.

Keep the outer try-catch that silently continues on any other exception (DB errors, etc.) -- this is correct behavior for a filter.

Add a logger (private val logger = LoggerFactory.getLogger) and log at INFO level when auto-creating a backend user: `"Auto-provisioned backend user for Better Auth email: {}"`.
