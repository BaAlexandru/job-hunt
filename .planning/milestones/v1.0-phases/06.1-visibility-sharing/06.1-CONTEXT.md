# Phase 6.1: Visibility & Sharing - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Users can control visibility of their companies and jobs as PRIVATE (default), PUBLIC, or SHARED with specific users. Includes share management endpoints (create/revoke by email), query logic for browsing public content, and merged list views that include shared-with-me items. Applications, interviews, and documents are NOT in scope — they remain private always.

</domain>

<decisions>
## Implementation Decisions

### Visibility scope
- Visibility applies to companies and jobs ONLY — not applications, interviews, or documents
- Visibility enum: PRIVATE (default), PUBLIC, SHARED
- Each entity's visibility is independent — no cascade from company to jobs
- Existing data migrated as PRIVATE via Flyway (DEFAULT 'PRIVATE' on new column)

### Sharing model
- Share via email lookup: owner enters email, system resolves to registered user. Error if email not found (no invite flow)
- Share grants stored in a join table with: owner_id, target_user_id, resource_type (COMPANY/JOB), resource_id, permission (VIEW/EDIT)
- Optional `includeJobs` flag when sharing a company — also creates share grants for all linked jobs
- Owner can revoke any share grant instantly via DELETE endpoint
- Only owner sees the share list — shared users cannot see other recipients

### Permissions
- VIEW (default): shared user can read the item, nothing else
- EDIT: shared user can update fields (name, description, etc.) but NOT delete, archive, or change visibility/sharing
- Delete and archive remain owner-only
- Owner retains all rights regardless of shares

### Public browse
- Browsing public content requires authentication (all /api/** routes require JWT — no SecurityConfig changes)
- New /api/public/companies and /api/public/jobs endpoints for browsing all public items from all users
- Public items show all the same fields as owner view — no redaction
- Public browse supports same pagination, search, and filters as existing list endpoints (name search, jobType, workMode)
- Public items include owner's display name (not email) for attribution

### List behavior (merged views)
- Existing /api/companies and /api/jobs endpoints return own items + shared-with-me items in a single list
- Each response item includes an `ownership` field: OWNER, SHARED_VIEW, or SHARED_EDIT
- Optional `?ownership=OWNER` or `?ownership=SHARED` query param to filter the merged list
- Default (no filter) shows all items (own + shared)
- Backward compatible: existing items all return ownership=OWNER

### Viewing shared/public companies and their jobs
- Viewing a shared/public company shows company details only
- Linked jobs are only visible if those jobs are themselves PUBLIC or SHARED with the viewer
- Per-entity visibility is respected — company visibility does not grant job visibility

### Migration & defaults
- New `visibility` column: VARCHAR(20) DEFAULT 'PRIVATE' on companies and jobs tables
- All existing rows automatically get PRIVATE — no data migration script needed
- New `share_grants` table with unique constraint on (target_user_id, resource_type, resource_id)
- Changing visibility from SHARED/PUBLIC to PRIVATE auto-deletes all share grants for that item
- FK ON DELETE CASCADE on share_grants for user deletion

### Claude's Discretion
- Exact DTO field naming and response structure details
- Repository query implementation (JPQL vs Spring Data naming)
- Index strategy for visibility column, share_grants lookups
- share_grants table column naming and constraint details
- Public browse endpoint path design (/api/public/* vs alternatives)
- How ownership enum is serialized in responses
- Error messages for share-to-unregistered-email

</decisions>

### Established Patterns
- Plain UUID foreign keys (no @ManyToOne) — use for share_grants table references
- `findByIdAndUserId` repository pattern — extend to also check visibility/share grants
- JPQL @Query with CAST(:param AS string) for nullable search parameters
- Batch resolution via findAllByIdInAndUserId for N+1 prevention
- Flyway: next migration sequence continues from Phase 6 (check latest V number)
- Pagination with Spring Pageable, default sort by createdAt descending

### Integration Points
- CompanyEntity + JobEntity: Add visibility VARCHAR column
- CompanyRepository + JobRepository: Modify list queries to include shared items, add ownership filter
- New ShareGrantEntity + ShareGrantRepository
- New ShareController for managing shares (create/list/revoke)
- New PublicBrowseController for /api/public/* endpoints
- CompanyService + JobService: Authorization logic changes (owner OR share grant check)
- Flyway: Migration to add visibility column + share_grants table
- SecurityConfig: No changes needed (public browse still requires auth)

## Specific Ideas

- User explicitly chose merged lists over separate /shared endpoints — shared items appear in the same list as owned items
- The `includeJobs` flag on company sharing was specifically requested for convenience — share a company and optionally its jobs in one call
- Attribution uses display name only (never email) for privacy
- The original deferred idea from Phase 3 ("Public/private visibility on companies and jobs") is now being implemented
